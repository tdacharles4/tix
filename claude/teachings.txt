9. TEST PLAN — BUG #9 (buyer quantity bounded)
────────────────────────────────────────────────────────────────────────────────
tests/unit/api/checkout/route.test.ts (cap = Math.min(session.max_quantity,
event.max_tickets_per_order)):
  1. quantity within cap → passes, proceeds to lockInventory.
  2. quantity exactly at cap → passes (boundary).
  3. quantity above cap → 400, Spanish error message.
  4. quantity = 0 or negative → 400.
  5. non-integer quantity (e.g. 2.5) → 400.
  6. KNOWN OPEN BUG — event.max_tickets_per_order is null (organizer left
     the field blank when creating the event): Math.min(session.max_quantity,
     null) coerces to 0, so cap = 0 and every quantity >= 1 is rejected,
     locking out checkout entirely. This test is EXPECTED TO FAIL until the
     null-cap lockout is actually fixed (e.g. defaulting the column in the
     DB, or guarding the Math.min call in code). Write it now as a
     known-red test so the fix can be verified against it later — do not
     silently skip it.

tests/unit/api/checkout/session/route.test.ts (POST — link generation):
  7. quantity <= event.max_tickets_per_order → session created, max_quantity
     stored correctly.
  8. quantity > event.max_tickets_per_order → 400 at link-generation time.
  9. event.status !== 'live' → 404 'Event not available'.
  10. SAME OPEN BUG, inverted effect: event.max_tickets_per_order is null →
      Number(quantity) > null coerces null to 0, so ANY quantity >= 1 is
      rejected at link-generation (organizer can't even generate a working
      checkout link for that event). Also expected to fail until fixed —
      pin down both manifestations of the same root cause.
