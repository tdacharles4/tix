BUG #3 — Race condition in ticket redemption (double-scan)
File: lib/qr/validate.ts (validateAndRedeemTicket), called from
      app/api/tickets/[id]/validate/route.ts — this is the live check-in path
      (app/checkin/[eventId]/page.tsx scans a QR and POSTs to
      /api/tickets/:id/validate, which redeems in the same call).
Note: app/api/tickets/redeem/route.ts was a second, parallel implementation of
      this same bug with zero live callers. It has been deleted as dead code —
      fix the version below, not that one.
────────────────────────────────────────────────────────────────────────────────

WHAT:
validateAndRedeemTicket checks if a ticket is active and then updates it to
redeemed in two separate database round trips. If two devices scan the same
QR code at exactly the same time, both can pass the status === 'active' check
before either write lands, and both will return valid: true — meaning the same
ticket gets accepted twice.

The live update has no .eq('status', 'active') guard at all — it only filters
on .eq('id', ticketId) — so it's worse than a partial guard: there is no check
on whether the update actually affected any rows, and nothing stops the write
from re-applying even if the row changed status between the SELECT and the
UPDATE. A silent or stale update still returns valid: true to the scanner.

WHY IT MATTERS:
At a busy event with multiple scanners, this is a realistic scenario. Someone
could screenshot their QR and let two people walk in on the same ticket.

HOW TO FIX:
Move the redemption logic into a Postgres RPC function that performs the check
and update atomically in a single statement:

  UPDATE tickets
  SET status = 'redeemed',
      redeemed_at = now(),
      redeemed_by = p_scanned_by
  WHERE id = p_ticket_id
    AND event_id = p_event_id        ← scope to event (also fixes security bug #4 prep)
    AND status = 'active'
  RETURNING *;

If RETURNING * gives back a row → redemption succeeded.
If it gives back nothing → ticket was already redeemed or not active.
The database handles the race at the transaction level — only one concurrent
call can win the update.

validateAndRedeemTicket (and the route that calls it) then simply calls the
RPC and branches on whether it returned a ticket row or null. No separate
SELECT needed.

BEFORE:
  SELECT ticket              ← round trip 1
  check status === 'active'  ← in application code (raceable)
  UPDATE ticket              ← round trip 2 (no status guard, no row-count check)
  return valid: true         ← even if update hit 0 rows or status changed mid-flight

AFTER:
  CALL redeem_ticket RPC     ← single atomic operation
  if returned row → valid: true
  if null → valid: false, reason: already_redeemed (or not_found)
