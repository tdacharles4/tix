MOCKING STRATEGY — Supabase
────────────────────────────────────────────────────────────────────────────────
Every route/lib file under test calls createServiceClient() or createClient()
from lib/supabase/server.ts. Unit tests should mock that module so no network
call ever reaches Supabase. The query patterns used across the codebase are
narrow and repeat constantly:

  .from(table).select(cols).eq(col, val)[.eq(col2, val2)].single()
  .from(table).update(obj).eq(col, val)
  .from(table).insert(arrOrObj)
  .rpc(fnName, args)

Build one small reusable fake client (e.g. test/fakeSupabase.ts) that returns
canned { data, error } per call, configured per test:

  function fakeSupabase(responses: { from?: Record<string, any>; rpc?: Record<string, any> }) {
    return {
      from: (table: string) => chainable(responses.from?.[table]),
      rpc:  (fn: string) => Promise.resolve(responses.rpc?.[fn] ?? { data: null, error: null }),
    };
  }
  // chainable() returns an object whose .select/.eq/.update/.insert/.single
  // all return `this` except the terminal call, which resolves the canned response.

vi.mock('@/lib/supabase/server', () => ({
  createServiceClient: () => fakeSupabase({ ... }),
  createClient: async () => fakeSupabase({ ... }),
}));

This tests the TypeScript branching logic, NOT whether Postgres itself
enforces atomicity — see next section.

────────────────────────────────────────────────────────────────────────────────
WHAT UNIT TESTS CAN'T COVER
────────────────────────────────────────────────────────────────────────────────
Bug #3's actual fix is a guarantee made by Postgres (UPDATE ... WHERE status =
'active' RETURNING * is atomic at the transaction level). A mocked Supabase
client can verify that validateAndRedeemTicket calls the RPC with the right
args and branches correctly on its response — it cannot prove two concurrent
calls can't both win. That requires an integration test against a real
Postgres (local `supabase start` stack or a disposable test schema): fire two
redeem_ticket RPC calls at the same ticket id concurrently (e.g. Promise.all)
and assert exactly one returns a row. Keep that as a separate
integration-test script, not part of the Vitest unit suite.

TEST PLAN — BUG #3 (redemption race condition)
────────────────────────────────────────────────────────────────────────────────
lib/qr/validate.ts — validateAndRedeemTicket(ticketId, eventId, redeemedBy):
  1. RPC returns a row → { valid: true, ticket, event, buyer }, hydrated from
     the follow-up select.
  2. RPC returns [] / null, fallback ticket.event_id !== eventId →
     { valid: false, reason: 'wrong_event' }.
  3. RPC returns [] / null, fallback status === 'cancelled' →
     { valid: false, reason: 'cancelled' }.
  4. RPC returns [] / null, fallback status === 'transferred' →
     { valid: false, reason: 'transferred' }.
  5. RPC returns [] / null, fallback status === 'redeemed' (event_id matches) →
     { valid: false, reason: 'already_redeemed' }.
  6. RPC returns [] / null, fallback select finds no ticket at all →
     { valid: false, reason: 'not_found' }.
  7. RPC call errors → function throws (don't swallow).
  8. Assert supabase.rpc('redeem_ticket', {...}) is called with exactly
     { p_ticket_id: ticketId, p_event_id: eventId, p_scanned_by: redeemedBy }
     — this is the test that would have caught the earlier swapped-argument
     mistake during implementation.

app/api/tickets/[id]/validate/route.ts:
  9.  No authenticated user → 401.
  10. validateAndRedeemTicket resolves → 200 with its result passed through.
  11. validateAndRedeemTicket throws → 500 with the error message.
  12. eventId missing from request body → confirm current behavior (it's
      passed straight through to validateAndRedeemTicket as undefined —
      decide if that should 400 instead; write the test either way so the
      behavior is pinned down, not implicit).

Integration-only (not Vitest): concurrent redeem_ticket RPC calls against a
real Postgres — exactly one of two simultaneous calls returns a row.

TEST PLAN — BUG #9 (buyer quantity bounded)
────────────────────────────────────────────────────────────────────────────────
app/api/checkout/route.ts (cap = Math.min(session.max_quantity,
event.max_tickets_per_order)):
  1. quantity within cap → passes, proceeds to lockInventory.
  2. quantity exactly at cap → passes (boundary).
  3. quantity above cap → 400, Spanish error message.
  4. quantity = 0 or negative → 400.
  5. non-integer quantity (e.g. 2.5) → 400.
  6. KNOWN OPEN BUG — event.max_tickets_per_order is null (organizer left the
     field blank when creating the event): Math.min(session.max_quantity,
     null) coerces to 0, so cap = 0 and every quantity >= 1 is rejected,
     locking out checkout entirely. This test is EXPECTED TO FAIL until the
     null-cap lockout is actually fixed (e.g. defaulting the column in the
     DB, or guarding the Math.min call in code). Write it now as a known-red
     test so the fix can be verified against it later — do not silently
     skip it.

app/api/checkout/session/route.ts (POST — link generation):
  7. quantity <= event.max_tickets_per_order → session created, max_quantity
     stored correctly.
  8. quantity > event.max_tickets_per_order → 400 at link-generation time.
  9. event.status !== 'live' → 404 'Event not available'.
  10. SAME OPEN BUG, inverted effect: event.max_tickets_per_order is null →
      Number(quantity) > null coerces null to 0, so ANY quantity >= 1 is
      rejected at link-generation (organizer can't even generate a working
      checkout link for that event). Also expected to fail until fixed —
      pin down both manifestations of the same root cause.

app/api/checkout/session/route.ts (GET — token validation):
  11. token not found / already used / expired → correct valid:false +
      reason string for each case.
  12. valid session → returns { valid: true, max_quantity, ticketTypeConfigId }.
